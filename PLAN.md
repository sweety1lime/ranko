# PLAN.md — Ranko

Веб-приложение, которое убивает 40 сообщений «ну а ты что хочешь?» в групповом чате.
Один человек создаёт вопрос с вариантами, кидает ссылку, все ранжируют варианты за 30 секунд
без регистрации и установки приложения, алгоритм находит честный компромисс.

Этот файл — источник правды для Claude Code. Работаем по фазам, строго по одной за сессию.

---

## 1. Продукт в одном абзаце

**Проблема:** выбор ресторана / фильма / места поездки в групповом чате превращается в бесконечный
пинг-понг, где побеждает самый настойчивый, а не лучший общий вариант.

**Решение:** страница-«решение» с вариантами → ссылка в чат → каждый перетаскивает варианты
в свой порядок предпочтений → живая страница результатов показывает компромисс-победителя
и расклад. Участникам не нужен аккаунт: открыл ссылку, ввёл имя, проголосовал.

**Анти-цели MVP (сознательно НЕ делаем):** регистрация участников, мобильные приложения,
опросы по датам, делёжка расходов, комментарии, уведомления. Это бэклог v0.2+ (раздел 10).

## 2. MVP-скоуп (v0.1)

Пользовательские истории:

1. **Создатель** на главной странице вводит: заголовок («Куда идём в пятницу?»),
   опционально описание, список вариантов (от 2 до 20), опционально дедлайн голосования.
2. После создания получает **две ссылки**: публичную для участников (`/d/{slug}`)
   и админскую (`/d/{slug}/admin?token={adminToken}`). Кнопки «скопировать» для обеих.
3. **Участник** открывает публичную ссылку: видит вопрос и варианты, вводит имя,
   перетаскивает варианты в порядок предпочтений (drag & drop или кнопки ↑/↓, сверху — самый желанный),
   жмёт «Отправить». Может вернуться по той же ссылке и изменить свой порядок, пока голосование открыто
   (его узнаём по токену в localStorage).
4. **Страница результатов** (`/d/{slug}/results`): текущий победитель, очки всех вариантов,
   кто уже проголосовал (имена), сколько участников. Обновляется сама (поллинг раз в 4 сек).
5. **Админ** может: закрыть голосование досрочно, удалить участника (спам/дубль), удалить решение.
   После закрытия страница голосования редиректит на результаты.
6. Ссылка красиво разворачивается в мессенджерах: **OG-превью** с заголовком вопроса.

Ограничения для простоты v0.1:
- Варианты **замораживаются** после первого голоса (нельзя добавлять/удалять — иначе ломаются ранжировки).
- Участник обязан ранжировать **все** варианты (список уже отсортирован по умолчанию — просто двигаешь).
- Никакой модерации контента, кроме лимитов длины полей.

## 3. Алгоритм компромисса

**Метод Борда** — простой и объяснимый пользователю:
- N вариантов; вариант на позиции k у участника получает `N − k` очков (топ-вариант = N−1, последний = 0).
- Победитель — максимум суммы очков.
- Тай-брейк: (1) больше первых мест → (2) меньше последних мест → (3) детерминированный
  случайный выбор с сидом = slug решения (чтобы результат не «прыгал» при обновлении).

Алгоритм — чистая функция `computeResults(options, rankings)` в `src/lib/borda.ts`,
покрытая юнит-тестами (включая тай-брейки и пограничные случаи: 0 и 1 участник).
На страницу результатов отдаём также «профиль» каждого варианта: сумма очков, число первых мест,
число последних мест — чтобы показать не только победителя, но и «почему».

## 4. Стек (не обсуждается внутри фаз — менять только правкой этого файла)

| Слой | Выбор | Почему |
|---|---|---|
| Фреймворк | Next.js (App Router) + TypeScript strict | один репозиторий на фронт и API, простой деплой |
| UI | Tailwind CSS + shadcn/ui | быстро и прилично из коробки |
| Drag & drop | dnd-kit + кнопки ↑/↓ как fallback | стандарт с тачем; но при списке до 20 вариантов на телефоне чистый драг конфликтует со скроллом — стрелки надёжнее и доступнее (a11y) |
| БД | Postgres (Neon, free tier) | serverless-совместимо; локально — Neon dev branch или Docker |
| Драйвер БД | Neon HTTP (`@neondatabase/serverless` + `drizzle-orm/neon-http`) | обязательно на Vercel: обычный TCP-`pg` исчерпает лимит соединений («too many connections») |
| ORM | Drizzle + drizzle-kit migrations | типобезопасно, лёгкий |
| Валидация | zod — на каждой границе API | |
| Данные на клиенте | SWR c `refreshInterval` для поллинга | realtime в MVP не нужен |
| ID | nanoid: slug (10 симв.), adminToken (24), participantToken (24) | |
| Тесты | Vitest (юниты) + Playwright (1 сквозной сценарий) | |
| Деплой | Vercel | нулевой ops |
| OG-превью | ImageResponse (`next/og`) | динамическая картинка с заголовком |

Аутентификации нет: право админа = знание adminToken, личность участника = participantToken
в localStorage + httpOnly-cookie дублем. Токены никогда не логируем.

## 5. Модель данных

```
decisions
  id            uuid PK
  slug          text unique      -- nanoid(10), для публичной ссылки
  admin_token   text             -- nanoid(24)
  title         text (1..120)
  description   text nullable (0..500)
  status        enum: 'open' | 'closed'
  deadline      timestamptz nullable
  created_at    timestamptz

options
  id            uuid PK
  decision_id   FK -> decisions (cascade delete)
  label         text (1..80)
  position      int              -- порядок при создании

participants
  id            uuid PK
  decision_id   FK -> decisions (cascade delete)
  name          text (1..40)
  token         text             -- nanoid(24), возвращается только владельцу
  created_at    timestamptz

rankings
  participant_id  FK -> participants (cascade delete)
  option_id       FK -> options
  rank            int            -- 0 = самый желанный
  PK (participant_id, option_id)
  UNIQUE (participant_id, rank)
```

Дедлайн проверяем лениво: любой read/write при `now > deadline` сначала переводит статус в `closed`.

## 6. API (route handlers, всё валидируется zod)

```
POST   /api/decisions
       body: { title, description?, options: string[], deadline? }
       → { slug, adminToken }                      (rate limit по IP, фаза 5)

GET    /api/decisions/{slug}
       → решение + варианты + участники (имена) + status
       (без admin_token и без чужих токенов!)

POST   /api/decisions/{slug}/participants
       body: { name } → { participantToken }       (403 если closed)

PUT    /api/decisions/{slug}/vote
       body: { participantToken, order: optionId[] }  -- полная перестановка всех вариантов
       создаёт/перезаписывает ranking; 403 если closed; 422 если order неполный

GET    /api/decisions/{slug}/results
       → { status, tally: [{optionId, label, points, firstPlaces, lastPlaces}],
           winnerId, participantsCount, votedNames[] }

POST   /api/decisions/{slug}/close      body: { adminToken }
DELETE /api/decisions/{slug}            body: { adminToken }
DELETE /api/decisions/{slug}/participants/{id}   body: { adminToken }
```

Правила: adminToken сравниваем через timing-safe compare; все ошибки — JSON `{ error }` с честным статусом.

## 7. Экраны

```
/                    лендинг = форма создания (заголовок, варианты с кнопкой «+ вариант», дедлайн)
/d/[slug]            голосование: имя (если ещё не участник) → drag-список → «Отправить»
                     после отправки — кнопка «К результатам» и «Изменить мой порядок»
/d/[slug]/results    победитель крупно, таблица очков, аватарки-инициалы проголосовавших, поллинг
/d/[slug]/admin      ?token=…: те же результаты + кнопки «Закрыть», «Удалить решение», удаление участников
```

Дизайн: mobile-first (основной сценарий — открыли из мессенджера на телефоне), одна акцентная
кнопка на экран, состояния пустоты и ошибок обязательны (протухшая ссылка, закрытое голосование).

## 8. Фазы для Claude Code

Правило: одна фаза = одна сессия. Каждую фазу начинать в **plan mode**, ревьюить план, потом
разрешать правки. В конце фазы: линт + тайпчек + тесты зелёные → коммит → `/clear`.

### Фаза 0 — Скелет
Задачи: создать Next.js-проект (TypeScript, Tailwind, ESLint), подключить shadcn/ui, Drizzle + Neon
(через HTTP-драйвер `drizzle-orm/neon-http`, не TCP-`pg` — иначе на Vercel исчерпаются соединения),
завести `.env.example` (`DATABASE_URL`), скрипты `dev/build/lint/typecheck/test`, пустой деплой на Vercel,
git-репозиторий, CLAUDE.md из раздела 9.
Готово, когда: `npm run dev` показывает заглушку; `npm run lint && npm run typecheck` зелёные; hello-world живёт на Vercel.

> Промпт: «Прочитай PLAN.md. Выполни Фазу 0. Ничего из других фаз не делай. Покажи план перед правками.»

### Фаза 1 — Данные и логика
Задачи: схема из раздела 5 + миграции; `src/lib/borda.ts` (`computeResults`) с юнит-тестами
(обычный случай, тай-брейк первыми местами, тай-брейк последними, детерминированный сид, 1 участник, 0 участников);
seed-скрипт с демо-решением.
Готово, когда: `npm run test` зелёный, `drizzle-kit` применяет миграции на чистую базу.

### Фаза 2 — API
Задачи: все ручки из раздела 6, zod-схемы в `src/lib/schemas.ts`, ленивое закрытие по дедлайну,
timing-safe сравнение adminToken. Интеграционные тесты на happy path + 403/422.
Готово, когда: сценарий «создать → присоединиться → проголосовать → результаты → закрыть» проходит тестами.

### Фаза 3 — Создание и голосование (UI)
Задачи: лендинг с формой создания и экраном «две ссылки»; страница `/d/[slug]`: ввод имени,
drag-ранжирование (dnd-kit, обязательно проверить touch) с кнопками ↑/↓ на каждой карточке
как fallback и a11y-режимом (та же перестановка, вторая точка входа), отправка,
повторный вход по participantToken из localStorage, изменение голоса.
Готово, когда: с телефона (эмуляция) можно пройти путь участника без единого тапа мимо
(и драгом, и стрелками).

### Фаза 4 — Результаты и админка
Задачи: `/d/[slug]/results` с поллингом SWR (4 сек), победитель + таблица очков + «почему победил»;
админ-страница с закрытием, удалением участника и решения; редирект голосования на результаты после закрытия.
Готово, когда: два окна браузера видят голоса друг друга без ручного обновления.

### Фаза 5 — Полировка
Задачи: OG-image через `next/og` (заголовок вопроса + число вариантов); rate limit на создание
(Upstash Ratelimit или простой вариант на Vercel KV); пустые состояния и страницы ошибок (404 по slug,
закрытое голосование); заморозка вариантов после первого голоса; финальная мобильная вёрстка.
Готово, когда: Playwright-сценарий (создать → 2 участника → результаты → закрыть) зелёный.

### Фаза 6 — Запуск
Задачи: прод-деплой, аналитика без cookies (Plausible или Umami), простая форма фидбека
(mailto или Tally), README с описанием и скриншотами.
Метрики успеха первых недель: создано решений; % решений, где проголосовали ≥ 3 человек;
% решений, доведённых до «закрыто».

## 10. Как работать с Claude Code (шпаргалка)

- Каждую фазу начинай в plan mode (Shift+Tab или /plan): Claude только читает и предлагает план,
  правки — после твоего одобрения.
- Одна фаза — одна сессия; после коммита делай /clear, чтобы контекст не «протухал».
- В конце фазы попроси ревью в свежем контексте: «Сабагентом проверь дифф этой фазы против PLAN.md
  и перечисли расхождения».
- Тесты — твой внешний оракул: заканчивай промпты фразой «прогони тесты и почини падения».
- Если Claude предлагает изменить стек или скоуп — это правка PLAN.md руками, а не решение в сессии.
